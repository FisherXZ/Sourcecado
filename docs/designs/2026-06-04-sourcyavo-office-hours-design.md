## Problem Statement

Codeology sourcing knowledge is scattered across Notion, Google Drive, Google Sheets, exported email threads, and possibly Messenger. Sourcing Directors need to reconstruct who Codeology contacted, who responded, who needs follow-up, who worked with the club before, and what worked across past semesters.

SourcyAvo should not be framed as a chatbot over documents. It is a sourcing-facing experience on top of a sourcing memory system: a permission-aware memory layer connected to a capable agent harness.

## What Makes This Cool

The exciting version is a real Club Brain for sourcing. A Sourcing Director can ask a messy question like "Who did we contact last semester for AI safety and who needs follow-up?" and get a direct answer with citations, missing-context warnings, and next actions.

The learning value is also part of the product. Members should learn the real shape of modern agent memory: source ingestion, structured records, embeddings, graph relationships, answer synthesis, citations, gap analysis, permission boundaries, and memory refresh.

## Constraints

- First user is the Sourcing Director.
- First interface is Research Chat.
- Dashboards, browsing panels, polished admin pages, and CRM-style workflows are deferred.
- The first Source Materials are Notion, Google Drive docs, Google Sheets, and exported email threads.
- Messenger is valuable because alumni and leadership context may live there, but it is a stretch source until access and privacy are resolved.
- The system should support three access tiers: Public Club Context, Officer Context, and Restricted Sourcing Context.
- Permission enforcement should happen at source/file separation and vector metadata filtering, not through model instructions alone.
- Security is required for trust, but it is not the product's top-level promise.

## Premises

1. SourcyAvo is a sourcing memory system connected to a powerful agent harness, not merely a chat product.
2. The wedge is answering real sourcing history questions better than a Sourcing Director manually searching Notion, Drive, Sheets, and email.
3. The teaching goal is part of the product; the architecture must be understandable enough for members to build and explain.
4. Security is a trust requirement, but the product promise is better sourcing memory and better sourcing decisions.
5. GBrain is a useful reference because several of its architectural primitives match current agent-memory best practice, but SourcyAvo should trim by selecting primitives, not by flattening the system into basic RAG.

## Landscape Note

The useful industry pattern is hybrid memory, not plain vector search. GBrain's public README emphasizes synthesized answers, citations, gap analysis, graph traversal, permission-scoped access, and a memory refresh cycle. Recent agent-memory research points in the same direction: Zep uses a temporal knowledge graph for dynamic business and conversation data, AriGraph combines semantic and episodic memory in a graph, and H-Mem combines temporal/semantic hierarchy with a knowledge graph.

The implication for SourcyAvo: keep the primitives that teach this memory-layer shape, but reduce their scale and autonomy for the club project.

## Approaches Considered

### Approach A: Minimal RAG Chat

Ingest Notion, Sheets, and docs into chunks, add embeddings, retrieve relevant chunks, and answer with citations.

Effort: S
Risk: Medium

Pros:
- Fastest to build.
- Easy to explain in a first demo.
- Good enough for basic source lookup.

Cons:
- Weak relationship modeling.
- Gap analysis is shallow.
- Permissions can become bolted on instead of built in.

### Approach B: Teachable Club Brain

Build a small but real memory architecture with source records, permission tiers, chunks, citations, Contact graph, outreach outcomes, and chat synthesis.

Effort: M
Risk: Medium

Pros:
- Preserves all five memory-layer pillars.
- Teaches current agent-memory architecture without cloning all of GBrain.
- Gives Sourcing Directors useful answers, not just search results.

Cons:
- More moving parts than a basic RAG demo.
- Requires careful curriculum sequencing.
- Needs disciplined scope control around integrations.

### Approach C: Full GBrain-Inspired Platform

Build a richer autonomous memory system with recurring enrichment, broad graph extraction, multi-source connectors, admin workflows, and long-running agent operations.

Effort: XL
Risk: High

Pros:
- Closest to the long-term vision.
- Could become broader Codeology infrastructure.
- Most directly mirrors the GBrain/OpenClaw/Hermes pattern.

Cons:
- Too much for MVP.
- Privacy and reliability risk rise quickly.
- Curriculum gets swallowed by integration and infrastructure work.

## Recommended Approach

Use Approach B: Teachable Club Brain.

This is the right middle. It is ambitious enough to include synthesis, citations, graph relationships, gap analysis, and memory refresh, while still being narrow enough to teach and ship as an internal Codeology project.

The key trim is not "remove the hard parts." The key trim is to keep the load-bearing memory primitives and reduce their scale:

- simple source records instead of full multi-system ingestion
- simple role tiers instead of per-person permissions
- Contact-centered graph instead of arbitrary ontology design
- manual or weekly memory refresh instead of a full autonomous daemon
- cited answer contract instead of broad agent autonomy

## Memory Primitives To Preserve

- **Source records**: trusted source items from Notion, Drive, Sheets, and email exports.
- **Chunks**: searchable units derived from source records.
- **Source citations**: references from answer claims back to source records.
- **Contacts**: people or organizations Codeology may want external relationships with.
- **Outreach attempts**: records that Codeology contacted a Contact for a reason.
- **Outreach outcomes**: responded, did not respond, needs follow-up, worth researching more, worked with Codeology, or not a fit.
- **Relationship graph**: Contact, Person, Company, Semester, Event, Project, Outreach Attempt, and Outreach Outcome relationships.
- **Knowledge gaps**: missing, stale, inaccessible, or unresolved context that the answer should surface.
- **Memory refresh**: manual or scheduled process that updates the memory system with newly available context.

## First User Experience

The MVP starts with Research Chat.

Every good answer should include:

1. **Answer**: direct synthesized answer in plain English.
2. **Evidence**: citations to allowed source material.
3. **Gaps**: missing, stale, inaccessible, or unresolved context.
4. **Next action**: follow up, research more, verify a missing thread, update an outcome, or draft outreach.

Example:

> We contacted three biotech companies in Spring 2026. Two responded, one did not. The strongest response was from X because they replied within a week and asked for more info.
>
> Sources: Spring 2026 Sourcing Sheet, exported email thread.
>
> Gaps: I do not see a final outcome for Y. Messenger may have more context.
>
> Next: Follow up with Y and verify whether X was added to the event planning thread.

## Out Of Scope For MVP

- Sending emails or messages automatically.
- Full Messenger integration.
- Per-person permissions.
- Polished CRM dashboard.
- Complete GBrain clone.
- Full codebase indexing.
- Broad all-member access.
- Autonomous outreach.

## Success Criteria

SourcyAvo succeeds if a Sourcing Director can ask 5-10 real sourcing questions and get useful, cited answers faster than manually searching Notion, Drive, Sheets, and email.

The first benchmark should include questions like:

- Who did we contact last semester for AI, biotech, startups, or design?
- Who responded?
- Who did not respond?
- Who needs follow-up?
- Which companies or people have worked with Codeology before?
- What outreach style worked?
- What did not work?
- Who should we look into more?

## Open Questions

- What exact Notion pages and Sheets will seed the first Source Materials?
- What email export format is easiest and safest for the club to provide?
- Which Messenger data, if any, can be exported without violating trust or privacy expectations?
- Should the first agent harness be Codex/Claude during development, or should the system explicitly target OpenClaw/Hermes-style operation from the start?
- Which name is canonical: Sourcecado or SourcyAvo?
